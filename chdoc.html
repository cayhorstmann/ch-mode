Compact HTML Markup

The Compact HTML markup syntax is a representation of XHTML in a less cluttered form that is easier to scan and edit than traditional HTML. The key idea is to write 〈em Hello〉 instead of <em>Hello</em>. This avoids the visual clutter of the end tags. The space after the start tag helps too. By using non-ASCII delimiters 〈…〉, there need not be any special rules for escaping commonly used characters. In Compact HTML markup, you can write 〈pre cout << "Hello";〉 instead of <pre>cout &lt;&lt; &quot;Hello&quot;;</pre>.

Markup Rules

  1. A tag starts with 〈 and ends with a matching 〉. That's a U+2329 (LEFT-POINTING ANGLE BRACKET) and a U+232A (RIGHT-POINTING ANGLE BRACKET).
  2. Immediately afterwards is the tag name, such as 
    〈em Hello〉
    for 
    <em>Hello</em>
  3. After the tag name may optionally be a period and a class name, such as 
    〈div.note A note...〉
    for 
    <div class='note'>A note...</div>
    You can have multiple class names such as 
    div.note.caution
    which is converted to 
    <div class='note caution'>
  4. If the tag name is omitted and there is a period after the opening bracket, the tag name is set to span. For example, 
    〈.hand John Hancock〉
    is 
    <span class='hand' John Hancock</span>
  5. After the tag name and any dotted class names, you can have an id prefixed by #: 
    〈img#hamster ...〉
    becomes 
    <img id='hamster' .../>
    Note that there may be no space before the #.
  6. If the tag name is omitted and there is a space after the opening bracket, then the tag name is set to code. For example, 〈 main〉 is <code>main</code>.
  7. The tag name (and optional class names) can be followed by attribute name/value pairs. For example, 
    〈img src=hamster.jpeg alt='A hamster'〉
    or 
    〈a href=http://horstmann.com My homepage〉
    There must be exactly one space before the attribute name and no spaces around the =. Use single quotes for attribute values that are empty or contain spaces. In the unlikely case that the element contents starts with what looks like a name/value pair, precede it by two spaces or a newline. For example, 
    〈code  x=1〉
    or 
    〈pre
x=1〉
  8. If an a element has an href value that doesn't start with a #, has no other attribute, and has no contents, the href attribute value becomes the contents. For example, 
    〈a href='http://horstmann.com'〉
    becomes 
    <a href='http://horstmann.com'>http://horstmann.com</a>
  9. If the 〈 is followed by a ! or ?, everything until the matching 〉 is copied verbatim, and surrounded by < >. This lets you include processing instructions, comments, and doctypes. For example, the top of your document can start out as 
    〈?xml version='1.0' encoding='UTF-8'?〉
〈!DOCTYPE html PUBLIC '-//W3C//DTD XHTML 1.0 Strict//EN'
      'http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd'〉
  10. If the 〈 is followed by a &, then everything up to the matching 〉 is copied verbatim and followed by a ;. This lets you enter entity references. For example, 
    〈&mdash〉
    becomes 
    &mdash;
    There are only two (rare) situations where you must use this feature. To include 〈 and 〉, use 〈&#9001〉 and 〈&#9002〉. To include a single quote in a quoted attribute, use 〈&apos〉. In all other cases, you need not worry about escaping. For example, a & in an attribute is correctly converted into &amp;. 
    〈a href='http://google.com?q=Alice & Bobs〈&apos〉s Restaurant'〉
    turns into 
    <a href='http://google.com?q=Alice &amp; Bob&apos;s Restaurant'>
  11. The XHTML void elements area, base, br, col, command, embed, hr, img, input, keygen, link, meta, param, source, track, wbr are self-closing. For example, 〈hr〉 is <hr/>, but 〈script src='slidy.js'〉 is <script src='slidy.js'></script>.

Rationale

  • Markdown has a noble goal:
    The idea is that a Markdown-formatted document should be publishable as-is, as plain text, without looking like it’s been marked up with tags or formatting instructions.
    But to achieve that goal, there are rules whose details are fiendishly complex. And as soon as the going gets rough, you reach the limitations of Markdown and have to use HTML.
  • What about AsciiDoc? It's much less limited than Markdown, but it's pretty complex.
  • So why not just author in HTML? It is too verbose, with its matching start and end tags. And it is a hassle that one has to escape characters that are fairly common in programming (& <).
  • Having (em Hello) instead of <em>Hello</em> makes perfect sense to any Lisp programmer, and the SXML notation follows this idea to its logical conclusion. It's a fine notation, but not very author-friendly.
  • Xah Lee has a related proposal that makes use of three delimiter pairs: 
        〔link 「rel “self” href “http://xahlee.org/emacs/blog.xml”」〕
  • How do you enter the brackets on your keyboard? If you use Emacs, then simply add the following to your .emacs: 
    (fset 'anglebrackets [?〈 ?〉 left]) 
(global-set-key (kbd "M-[") 'anglebrackets)
    Then typing M-[ yields a pair of brackets, with the cursor in between. Alternatively, if you use Linux, make a ~/.XCompose file with this content: 
    include "%L"
<Multi_key> <bracketleft>        : "〈"   U2329 # LEFT-POINTING ANGLE BRACKET
<Multi_key> <bracketright>       : "〉"   U232A # RIGHT-POINTING ANGLE BRACKET
    Then you can use Compose [ and Compose ] to enter the brackets. I map my Compose key to Caps Lock, which makes this quite convenient.
  • How do you match the brackets? Emacs will match any kind of Unicode brackets, and if your text editor is any good, it will do the same.

So, there you have it. An XML entry format, optimized for XHTML, that removes the tedium of end tags and provides convenient shortcuts for common HTML constructs (classes, spans, ids, code, and character references). An automatic translator converts between this format and XML.
